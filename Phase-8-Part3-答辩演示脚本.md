### 🔴 必答题（90% 概率被问）

#### Q1：你这个项目最核心的技术难点是什么？

**回答模板**：

> "最核心的难点是**音频流式播放**。
>
> FLAC 是无损格式，单文件 30-80MB。浏览器原生 `<audio>` 标签不支持 FLAC 直接播放，更不能等整个文件下载完才播放。
>
> 我的解决方案是 **HTTP Range Request**——前端请求 `Range: bytes=0-65535` 头，后端 Spring Boot 的 ResourceHttpRequestHandler 自动响应 `206 Partial Content`，只传输请求的字节范围。
>
> 浏览器配合 `wavesurfer.js` 解码 PCM 后用 Web Audio API 播放——边下载、边解码、边播放。
>
> 配合 peaks 接口的预计算（后端用 FFmpeg 提取波形点 → JSON 序列化 → Redis 缓存），实现了**首次 750ms，缓存命中 130ms**的响应速度。"

**讲完这个，老师 80% 不会再问技术问题——这是 6 分答辩分里最实的 3 分。**

---

#### Q2：数据库设计你怎么考虑的？

**回答模板**：

> "数据库设计了 **11 张表**：
>
> 业务实体：用户、艺术家、专辑、曲目、歌单、标签
>
> 关系表：歌单-曲目、曲目-标签
>
> 行为表：播放历史、用户收藏、转换任务
>
> 设计原则：
>
> - **第三范式**：每个表单一职责，比如曲目表只放曲目属性，专辑信息走 album_id 外键关联
> - **软删除**：业务实体加 is_deleted + deleted_at 字段，MyBatis-Plus 自动过滤，**避免误删导致数据丢失**
> - **关系表硬删除**：playlist_track 这种纯关系不需要软删
> - **多表 JOIN 查询**：统计页的 Top 艺术家 = `JOIN track ON play_history` + GROUP BY，避免 N+1 查询
> - **字段命名带单位**：duration_seconds、file_size_bytes，**自描述**避免歧义
> - **utf8mb4 字符集**：支持 emoji 和日韩文字（库里有日文歌名'ナチュラル'）"

#### Q5：你怎么处理上传文件的元数据？

**回答模板**：

> "用 Java 的 **jaudiotagger** 库。
>
> 用户上传 FLAC/MP3 文件，后端 Spring Boot 接收 MultipartFile：
>
> 1. **生成 UUID 文件名**避免冲突，存到 `D:/soundprint-storage/audio/`
> 2. **jaudiotagger 读取 ID3 标签**：title、artist、album、duration、bitrate
> 3. **提取嵌入封面**到 storage/cover/
> 4. **自动创建艺术家/专辑**（如果数据库不存在）
> 5. 入库 track 记录
>
> 这样用户只需要**拖文件**，所有元数据自动填充——零手工录入。
>
> 失败处理：如果元数据缺失，前端用**银白渐变 + 名字首字**生成占位封面，**保证视觉一致性**。"

#### Q6：FFmpeg 怎么调用的？

**回答模板**：

> "Java 的 **ProcessBuilder** 启动子进程：
>
> ```java
> ProcessBuilder pb = new ProcessBuilder(
>     'ffmpeg', '-i', inputPath,
>     '-b:a', '320k',
>     '-ar', '44100',
>     '-y', outputPath
> );
> Process p = pb.start();
> ```
>
> 关键的实时进度推送：
>
> 1. FFmpeg 的进度输出在 **stderr**（不是 stdout）
> 2. 起一个线程读 stderr 流，**正则匹配 `time=00:01:23.45`**
> 3. 算出当前进度百分比
> 4. 通过 SSE 推送给前端
>
> 任务用 `@Async` 异步执行，不阻塞主线程。
>
> 完成后状态置 SUCCESS，前端轮询接口拿结果。"
